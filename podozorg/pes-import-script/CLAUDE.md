# CLAUDE.md

## Guiding principle

Code must be readable and modifiable by a human. Every function should do one thing. Prefer explicit, named types over inline types. If a new field is added, the developer should only need to change one or two places — not trace through the entire codebase.

## TypeScript runtime

- Scripts are executed with `tsx` (full TypeScript support including enums and decorators).
- Import enums directly from `src/shared/generated.ts` — this file is generated by `npm run codegen` in the `iac` directory.
- Never re-export enums as UPPER_SNAKE_CASE constants (e.g. `const SIDE = ESide`). Import and use them directly where needed.
- To validate a value against an enum, use `Object.values(EMyEnum).includes(value)`.

## Script structure

Each migration module follows this folder structure:

```
[module]/
├── [module].ts              # Main script: orchestration + logging
├── [module].config.ts       # Constants, paths, regex, CSV headers
├── helpers/
│   └── [module].helpers.ts  # All logic: filter → map → write
└── interfaces/
    └── [module].interface.ts # All interfaces and types
```

## Function separation

Helpers follow a three-stage pipeline:

- **`filterValid*()`** — validates input, returns valid items + skip counts. Uses early returns with clear skip reasons.
- **`map*ToCsvRows()`** — transforms valid items into CSV row strings. Pure mapping, no validation.
- **`write*Csv()`** — orchestrates filter → map → file write. Thin function, no business logic.

The main script (`[module].ts`) only calls the write function, logs results, and writes a log file.

## Config file

All paths, magic numbers, regex patterns, batch sizes, and CSV headers go in the config file. No hardcoded values in helpers or main scripts.

## Interfaces over inline types

All function return types and intermediate data shapes must be named interfaces in the interfaces file. Do not use inline `{ foo: string; bar: number }` as return types — extract them to a named interface.

## CSV parsing

- Use the `csv-parser` package with `{ separator: ';' }` for reading CSV files. Do not manually parse CSV with `split('\n')` / `split(';')`.

## Code style

- Prefer early return pattern over long `&&`-chains or deeply nested conditionals. Each guard clause should return early with a clear condition.
- Avoid iterating over the same collection multiple times. When multiple results are derived from the same data, combine them into a single loop.

## DynamoDB import scripts (iac/scripts)

Use `Object.keys()` to iterate over fields dynamically instead of manually listing every field. Define exceptions explicitly:

- **`NUMERIC_FIELDS`** — set of field names that need `parseNumber()` conversion.
- **`VALUE_TRANSFORMERS`** — record of field-specific transformers (JSON parse, boolean, optional string).
- **Default** — all other fields pass through as string.

When adding a new field: add it to the interface. If it's numeric, add to `NUMERIC_FIELDS`. If it needs special handling, add to `VALUE_TRANSFORMERS`. Nothing else needs to change.

## Naming conventions

- Components & Files: kebab-case (e.g. `user-profile.component.ts`)
- Test files: kebab-case with `.test.ts` suffix (e.g. `get-consult-by-id.test.ts`)
- Folders: kebab-case (e.g. `auth-guards`)
- Classes & Interfaces: PascalCase (e.g. `export class UserProfile`)
- Variables, Functions & Properties: camelCase (e.g. `getUserData()`)
- Constants: UPPER_SNAKE_CASE (e.g. `API_ENDPOINT`)
